# SECURITY.md — MebelLegal KZ

## Принципы безопасности

### 1. Deny-by-default

- по умолчанию доступ запрещён;
- доступ предоставляется явным правилом;
- отсутствие правила = отсутствие доступа.

### 2. Least privilege

- каждая роль получает минимально необходимые права;
- скрытие кнопки в интерфейсе не является контролем доступа;
- серверная авторизация обязательна для каждой команды.

### 3. Defence in depth

- аутентификация (Supabase Auth);
- авторизация (проверка роли на сервере);
- изоляция строк (RLS по `organization_id`);
- валидация входных данных (Zod);
- логирование (audit events).

### 4. Tenant isolation

- `organization_id` присутствует во всех пользовательских доменных сущностях;
- доступ проверяется сервером и базой данных;
- RLS включён для каждой пользовательской таблицы;
- политика по умолчанию — запрет;
- пользователь без активного membership не получает доступ;
- подмена UUID, URL или `organization_id` не меняет права;
- service role не попадает в браузер.

### 5. Безопасность секретов

- секреты хранятся только в защищённом окружении;
- секреты не попадают в Git, логи и клиентский bundle;
- `.env*`, кроме `.env.example`, не отслеживаются;
- service role используется только серверных компонентах.

### 6. Защита персональных данных

- PII не попадает в логи;
- payload audit event не содержит ИИН, БИН, телефоны, адреса, банковские счета;
- тесты используют синтетические данные;
- реальные договоры не коммитятся в Git.

### 7. Маскирование ошибок

- клиентский ответ не содержит stack trace;
- ошибки маскируются для внешнего наблюдателя;
- детали ошибок логируются серверно.

### 8. Rate limiting

- публичные endpoints имеют rate limiting;
- повторные запросы обрабатываются контролируемо.

## Среды

| Среда | Характеристика |
|---|---|
| `demo` | Только синтетические данные, внешние действия заблокированы |
| `pilot` | Ограниченные реальные данные одной/нескольких утверждённых компаний |
| `staging` | Тест релиза и миграций |
| `production` | Только после security/compliance gates |

## Резервное копирование

- резервное восстановление проверяется до реального пилота;
- правила срока хранения и удаления данных определяются до пилота.

## Тесты безопасности

Обязательные security-тесты:

- пользователь A не читает кейсы организации B;
- пользователь A не изменяет кейс организации B, даже подменив UUID;
- observer не создаёт и не изменяет кейс;
- manager не утверждает кейс;
- disabled membership не предоставляет доступ;
- пользователь без membership не видит организацию;
- audit events другой организации недоступны;
- прикладной пользователь не обновляет и не удаляет audit event.

## Threat model

- определён до запуска публичного согласования;
- contract tests для интеграций;
- проверка резервного восстановления до пилота.
