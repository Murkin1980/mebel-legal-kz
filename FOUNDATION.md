# FOUNDATION — MebelLegal KZ

**Статус:** обязательный архитектурный договор проекта  
**Расположение:** корень репозитория  
**Версия:** 1.0  
**Дата:** 20 июля 2026 года  

---

## 1. Назначение файла

Этот файл фиксирует фундамент MebelLegal KZ: границы продукта, правила владения данными, безопасность, версионность, роль AI и технические ограничения.

Каждый человек или AI-кодер обязан прочитать `FOUNDATION.md`:

1. до анализа новой задачи;
2. до проектирования изменения;
3. перед завершением работы;
4. перед изменением архитектуры, схемы данных или интеграционного контракта.

Если задача противоречит этому файлу, кодер должен остановиться, описать конфликт и запросить решение владельца. Нельзя обходить фундамент молча.

## 2. Миссия продукта

MebelLegal KZ — правовой контур для мебельного бизнеса Республики Казахстан.

Система должна помогать:

- подготовить комплект документов мебельного заказа;
- проверить комплектность и известные риски;
- связать вывод с утверждённым официальным источником;
- зафиксировать конкретную версию договора, спецификации и эскиза;
- получить и доказуемо сохранить решение клиента;
- контролировать изменения заказа;
- передать утверждённые данные в соседние системы.

Система помогает подготовке и контролю. Она не заменяет юриста, не гарантирует законность документа и не обещает исход судебного спора.

## 3. Главная архитектурная модель

На старте используется **модульный монолит**.

Микросервисы, отдельные очереди, второй backend и дополнительные хранилища добавляются только после доказанной необходимости.

Основной поток будущего продукта:

```text
источник заказа
→ юридический кейс
→ проверка данных
→ проверка правилами
→ версия договорного пакета
→ ручное утверждение
→ согласование клиентом
→ неизменяемая история
→ передача утверждённых данных
```

## 4. Границы трёх систем

### 4.1. MebelLegal KZ владеет

- реестром правовых источников и редакций;
- правилами юридической и документальной проверки;
- версиями шаблонов;
- юридическим кейсом мебельного заказа;
- договорным комплектом;
- результатами проверок;
- согласованием конкретной версии;
- изменениями заказа;
- претензионным контуром;
- юридическим audit log.

### 4.2. Interactive KP владеет

- коммерческим предложением;
- позициями и вариантами комплектации до импорта;
- маркетинговыми статусами КП;
- публичным выбором коммерческого варианта;
- коммерческими расчётами до фиксации юридического кейса.

### 4.3. MebelDocs AI владеет

- счетами;
- платежами;
- бухгалтерскими и закрывающими документами;
- З-2 и Р-1;
- ЭСФ;
- банковской сверкой;
- бухгалтерским архивом.

### 4.4. Запреты

- ни одна система не обращается напрямую к таблицам другой;
- ни одна система не становится скрытым владельцем чужого домена;
- данные передаются через версионированные API-контракты;
- внешние команды должны быть аутентифицированы, авторизованы и идемпотентны;
- сбой интеграции не должен повреждать исходные данные.

## 5. Мультитенантность

Любые пользовательские данные принадлежат конкретной организации.

Обязательные правила:

- `organization_id` присутствует во всех пользовательских доменных сущностях;
- доступ проверяется сервером и базой данных;
- RLS включён для каждой пользовательской таблицы;
- политика по умолчанию — запрет;
- пользователь без активного membership не получает доступ;
- подмена UUID, URL или `organization_id` не меняет права;
- service role не попадает в браузер;
- межтенантная изоляция подтверждается автоматическими security-тестами.

Новая таблица с пользовательскими данными без RLS считается незавершённой и не может быть опубликована.

## 6. Роли и минимальные привилегии

Базовые роли:

- `owner`;
- `manager`;
- `designer`;
- `legal_reviewer`;
- `observer`.

Права выдаются по принципу минимальной необходимости.

Скрытие кнопки в интерфейсе не является контролем доступа. Каждая команда проверяет роль на сервере, а доступ к строкам ограничивает база данных.

Только уполномоченная роль может:

- утверждать карточку кейса;
- публиковать юридическое правило;
- публиковать версию шаблона;
- подтверждать результат юридической проверки;
- отзывать клиентский запрос согласования.

## 7. Юридический кейс

`legal_case` — центральный контейнер правового процесса одного мебельного заказа.

Он не является:

- CRM-сделкой;
- коммерческим предложением;
- бухгалтерской реализацией;
- ЭСФ;
- производственным заказом.

Кейс хранит минимальную юридическую проекцию заказа и связи с внешними идентификаторами.

В будущем к кейсу подключаются:

- стороны;
- согласованные условия;
- позиции и выбранные варианты;
- вложения;
- проверки;
- версии договорного пакета;
- согласования;
- изменения;
- претензии;
- интеграционные события.

Не допускается хранить всю внешнюю сущность Interactive KP или MebelDocs AI в одном неуправляемом JSON.

## 8. Деньги

Деньги хранятся целым количеством тийинов.

```text
390 000 ₸ = 39 000 000 тийинов
```

Обязательные правила:

- JavaScript `number` и `float` не используются как доменное хранилище денег;
- базовый тип — `bigint`/целое значение + валюта;
- валюта MVP — KZT;
- арифметика детерминирована;
- правило округления процентов указывается явно;
- суммы цифрами и прописью создаются из одного источника;
- аванс + остаток должны совпадать с итогом;
- расчёт на клиенте не является авторитетным без серверной проверки.

Обязательный регрессионный тест:

```text
390 000 ₸ × 70% = 273 000 ₸
390 000 ₸ × 30% = 117 000 ₸
273 000 ₸ + 117 000 ₸ = 390 000 ₸
270 000 ₸ + 117 000 ₸ ≠ 390 000 ₸
```

## 9. Доменные команды

UI, API и AI не изменяют доменные таблицы произвольными запросами.

Изменения выполняются через доменные команды:

```text
command
→ authentication
→ authorization
→ validation
→ tenant check
→ state transition
→ transaction
→ audit event
→ result
```

Каждая юридически значимая команда:

- имеет стабильное имя;
- проверяет права;
- валидирует вход;
- поддерживает optimistic concurrency, где требуется;
- имеет `command_id` или `idempotency_key`;
- выполняется транзакционно;
- создаёт audit event;
- безопасно обрабатывает повтор запроса.

## 10. Статусы и state machines

Статус нельзя менять произвольной строкой.

Для каждой доменной сущности определяется state machine:

- разрешённые переходы;
- запрещённые переходы;
- требуемая роль;
- необходимые предварительные условия;
- создаваемое событие;
- возможность отмены или создания новой версии.

Статус `approved` карточки кейса не означает, что договор создан, проверен юристом, подписан или согласован клиентом. Разные виды утверждения должны иметь разные сущности и события.

## 11. Версионность и неизменяемость

Следующие данные после публикации или подтверждения не перезаписываются:

- редакция правового источника;
- опубликованное правило;
- опубликованная версия шаблона;
- сформированный договорный пакет;
- согласованный эскиз;
- подтверждение клиента;
- audit event.

Исправление создаёт новую версию с указанием предыдущей.

Юридически значимая версия хранит:

- номер версии;
- время создания;
- автора;
- хеш содержимого;
- используемые версии правил и источников;
- статус;
- ссылку на заменённую версию;
- события публикации, согласования и отзыва.

## 12. Audit log

Audit log является append-only.

Запрещено:

- изменять событие;
- удалять событие прикладным пользователем;
- переписывать историю при исправлении;
- хранить в payload полный договор или избыточные персональные данные.

Исправление или отмена фиксируется новым событием.

Audit event содержит минимум:

- организацию;
- действующего пользователя/систему;
- тип события;
- тип и ID сущности;
- command ID;
- время;
- безопасный payload;
- correlation ID;
- связь с предыдущим событием, если применимо.

## 13. Идемпотентность

Повтор одного запроса не создаёт вторую юридически значимую операцию.

Идемпотентность обязательна для:

- импорта внешнего заказа;
- создания версии пакета;
- публикации;
- решения клиента;
- изменения заказа;
- экспорта в MebelDocs AI;
- фоновых проверок источников.

Повтор команды с тем же ключом возвращает результат первоначального выполнения либо контролируемый конфликт, но не дублирует запись.

## 14. Правовые источники

Когда модуль будет разрешён отдельным этапом:

- каноническим источником является официальная страница ИПС «Әділет»;
- действует allowlist конкретных документов;
- бесконтрольный обход сайта запрещён;
- сохраняются ссылка, название, редакция, даты действия, время получения и хеш;
- старая редакция не заменяется новой;
- недоступность источника показывается пользователю;
- вывод без прямой нормы помечается как вывод или внутреннее правило;
- новое или изменённое правило публикует человек с соответствующей ролью.

До отдельного разрешения подключение и парсинг ИПС «Әділет» запрещены.

## 15. Шаблоны документов

`Договор_изготовление_мебели_B2B_шаблон_для_юриста.docx` является проектом для юридической проверки, а не опубликованным шаблоном.

До письменного утверждения юристом запрещено:

- автоматически отправлять его клиентам;
- маркировать его как юридически проверенный;
- использовать формулировки как обязательные правовые правила;
- делать его единственной формой для всех типов клиентов;
- переносить в production registry опубликованных шаблонов.

Будущий шаблон обязан иметь:

- тип клиента и сделки;
- версию;
- статус `draft`, `expert_review`, `published`, `retired`;
- автора и проверяющего;
- перечень переменных;
- условные блоки;
- связанные правила;
- дату действия;
- тестовые примеры;
- журнал публикации.

## 16. Искусственный интеллект

AI — помощник, а не источник истины и не субъект юридического решения.

AI разрешён в будущих этапах для:

- извлечения полей;
- классификации;
- поиска по утверждённому корпусу;
- объяснения результата правила;
- сравнения редакций;
- подготовки черновика формулировки;
- подготовки черновика ответа на претензию.

AI запрещён как единственный механизм для:

- сумм и процентов;
- сроков;
- прав доступа;
- state transitions;
- выбора применимой редакции без детерминированной проверки даты;
- публикации нормы, правила или шаблона;
- подписания и отправки;
- признания или отклонения претензии;
- изменения подтверждённого пакета;
- удаления истории.

Контур AI:

```text
AI proposal
→ источник и confidence
→ проверка пользователем
→ доменная команда
→ audit event
```

На первом этапе AI отсутствует.

## 17. Персональные данные и файлы

- реальные договоры, ИИН, БИН, адреса, телефоны и банковские реквизиты не коммитятся в Git;
- тесты используют синтетические данные;
- оригиналы документов хранятся приватно;
- постоянные публичные URL запрещены;
- секреты не попадают в Git, логи и клиентский bundle;
- логи не содержат PII;
- загрузки в будущем проходят проверку типа, размера и безопасности;
- применяется минимизация данных, передаваемых в AI;
- должны существовать правила срока хранения, удаления и резервного копирования до пилота.

## 18. Безопасность

Обязательные принципы:

- deny-by-default;
- least privilege;
- defence in depth;
- server-side authorization;
- RLS;
- private storage;
- секреты только в защищённом окружении;
- маскирование ошибок;
- rate limiting для публичных endpoints;
- короткоживущие и отзывные ссылки;
- threat model до запуска публичного согласования;
- contract tests для интеграций;
- резервное восстановление проверяется до реального пилота.

Фича без security-тестов для своих границ считается незавершённой.

## 19. Среды

Минимальные среды:

- `demo` — только синтетические данные, внешние действия заблокированы;
- `pilot` — ограниченные реальные данные одной/нескольких утверждённых компаний;
- `staging` — тест релиза и миграций;
- `production` — только после security/compliance gates.

Нельзя использовать production как тестовую среду.

## 20. Этапность

Проект развивается вертикальными этапами.

Текущий первый этап включает только:

- репозиторий и проектные документы;
- организации;
- memberships и роли;
- RLS;
- минимальный legal case;
- Money;
- доменные команды;
- state machine;
- append-only audit log;
- идемпотентность;
- минимальный внутренний UI;
- автоматические тесты.

До отдельного разрешения запрещены:

- ИПС «Әділет»;
- AI;
- генерация документов;
- публичное согласование;
- претензии;
- live-интеграции;
- реальные реквизиты;
- production deployment с реальными данными.

## 21. Требования к изменениям

Перед началом задачи кодер обязан ответить:

1. Какой модуль изменяется?
2. Какой проект владеет этими данными?
3. Какие пункты `FOUNDATION.md` применимы?
4. Затрагиваются ли tenant isolation, роли, деньги, версии, audit или AI?
5. Какие тесты докажут соблюдение фундамента?

Если изменение затрагивает фундаментальный принцип, нужен ADR и явное одобрение владельца.

## 22. Foundation Check перед завершением

В конце каждой задачи кодер добавляет в `SESSION_NOTES.md`:

```markdown
### Foundation Check

- [ ] Границы MebelLegal KZ / Interactive KP / MebelDocs AI не нарушены.
- [ ] Tenant isolation и RLS сохранены/проверены.
- [ ] Серверная авторизация присутствует.
- [ ] Деньги не хранятся в float/JavaScript number.
- [ ] State transitions выполняются доменной командой.
- [ ] Юридически значимые действия идемпотентны.
- [ ] Audit log дополнен и остаётся append-only.
- [ ] Подтверждённые данные не перезаписываются, а версионируются.
- [ ] AI не выполняет запрещённые решения.
- [ ] В Git, логах и fixtures нет реальных данных и секретов.
- [ ] Добавлены unit/integration/security/contract tests по риску изменения.
- [ ] Изменение соответствует разрешённому текущему этапу.
```

Невыполненный пункт должен быть объяснён как блокер или явно утверждённое исключение.

## 23. Изменение самого фундамента

`FOUNDATION.md` может изменяться, но не автоматически и не как побочный эффект фичи.

Процедура:

1. Описать проблему и причину изменения.
2. Создать ADR.
3. Указать затронутые модули, данные и риски.
4. Подготовить миграцию и обратимость, если требуется.
5. Получить явное одобрение владельца.
6. Изменить версию и дату `FOUNDATION.md`.
7. Обновить тесты и связанную документацию.
8. Зафиксировать решение в `SESSION_NOTES.md`.

Запрещено ослаблять безопасность, аудит, версионность или границы систем только ради ускорения разработки.

## 24. Неподлежащие компромиссу правила

1. Нет прямого доступа к таблицам соседнего проекта.
2. Нет пользовательской таблицы без tenant isolation и RLS.
3. Нет денег во float/JavaScript number.
4. Нет тихого изменения или удаления юридически значимой истории.
5. Нет публикации правового правила или шаблона без человека.
6. Нет AI как единственного механизма юридического решения.
7. Нет реальных договоров и реквизитов в Git.
8. Нет service role в браузере.
9. Нет повторной юридически значимой операции при повторном запросе.
10. Нет перехода к следующему этапу без приёмки текущего.
